# 全系统单号规则文档

## 设计原则

业务单号采用统一规则：

```text
YYYY + 前缀 + 6位流水
```

示例：

```text
2026CGDD000001
```

说明：

- `YYYY`：单据年份
- `前缀`：区分业务类型
- `6位流水`：当年从 `000001` 开始递增

## 为什么采用这套规则

- 一眼可识别单据类型
- 一眼可识别单据年份
- 人工查单、对账、打印更方便
- 不暴露系统主键
- 规则简单，前后端都容易理解

## 主键与单号分离

所有业务表建议同时保留两套标识：

- `id`
  - 雪花算法主键
  - 数据库 `BIGINT`
  - 接口按 `string` 返回
- `xxxNo`
  - 业务单号
  - 用于列表展示、搜索、打印、对账、沟通

示例：

- 采购订单
  - `id`
  - `orderNo`
- 采购入库
  - `id`
  - `inboundNo`

## 单号生成规则

### 通用规则

- 每种单据类型独立编号
- 每年单独从 `000001` 开始
- 单号由后端统一生成
- 单号字段必须加唯一索引
- 单号生成时间建议以服务端时间为准

### 流水号规则

- 长度固定为 6 位
- 不足 6 位左侧补 `0`

示例：

```text
000001
000002
000128
001024
```

如果后期某类单据年单量超过 `999999`，再升级为 7 位流水。

## 字段命名建议

建议命名统一，不要混乱：

- 订单类：`orderNo`
- 入库类：`inboundNo`
- 出库类：`outboundNo`
- 核准类：`approveNo`
- 物流类：`logisticsNo`
- 收款类：`receiptNo`
- 付款类：`paymentNo`
- 合同类：`contractNo`
- 对账类：`statementNo`

## 示例

### 采购订单

```json
{
  "id": "1914876201459236001",
  "orderNo": "2026CGDD000001"
}
```

### 销售出库

```json
{
  "id": "1914876201459236002",
  "outboundNo": "2026XSCK000001"
}
```

### 付款单

```json
{
  "id": "1914876201459236003",
  "paymentNo": "2026FKD000001"
}
```

## 注意事项

- 不要使用雪花 `id` 直接作为业务单号
- 不要把供应商名、客户名、项目名拼进单号
- 不要让前端生成正式单号
- 不要多个单据类型共用同一套流水

## 结论

全系统业务单号统一采用：

```text
YYYY + 前缀 + 6位流水
```

这是当前阶段按你要求统一后的正式规则。
