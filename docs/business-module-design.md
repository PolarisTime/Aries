# 业务模块设计文档

## 设计目标

本版模块设计用于钢材业务系统前端重构，当前覆盖以下核心模块：

- 采购订单
- 采购入库
- 销售订单
- 销售出库
- 价格核准
- 物流单
- 商品库存报表
- 出入库报表
- 供应商对账单
- 客户对账单
- 物流对账单
- 财务管理
  - 收款单
  - 付款单
  - 应收应付
- 合同管理

设计原则：

- 主数据和业务单据分离
- 单据链路清晰，避免重复录入
- 财务、对账基于业务单据汇总，不直接手工造数
- 模块边界明确，方便后续做菜单、权限、路由和页面拆分
- 采购、销售表单默认除隐藏 `id` 外，其余业务字段全部展示

## 模块总览

| 模块 | 作用 | 核心单据/页面 | 上游 | 下游 |
| --- | --- | --- | --- | --- |
| 采购订单 | 向供应商下采购需求 | 采购订单列表、详情、编辑 | 商品、供应商、合同、价格核准 | 采购入库、供应商对账、付款 |
| 采购入库 | 记录实际到货和入库 | 入库单列表、详情、入库明细 | 采购订单 | 库存、供应商对账、应付 |
| 销售订单 | 对客户形成销售需求 | 销售订单列表、详情、编辑 | 商品、客户、项目、合同、价格核准 | 销售出库、客户对账、收款 |
| 销售出库 | 记录实际发货出库 | 出库单列表、详情、出库明细 | 销售订单、物流单 | 库存、客户对账、应收 |
| 价格核准 | 控制成交价格或结算价 | 价格申请单、价格核准单 | 商品、客户、供应商 | 采购订单、销售订单 |
| 物流单 | 记录发运和承运信息 | 物流单列表、详情 | 销售出库或采购入库 | 物流对账、运费应付 |
| 商品库存报表 | 查库存结存 | 库存汇总、库存明细 | 入库、出库 | 业务查询 |
| 出入库报表 | 查流水 | 入库流水、出库流水 | 入库、出库 | 对账、审计 |
| 供应商对账单 | 汇总采购与付款 | 供应商对账页面 | 采购订单、采购入库、付款单 | 应付确认 |
| 客户对账单 | 汇总销售与收款 | 客户对账页面 | 销售订单、销售出库、收款单 | 应收确认 |
| 物流对账单 | 汇总运输和运费 | 物流对账页面 | 物流单、付款单 | 运费结算 |
| 收款单 | 记录客户收款 | 收款单 | 客户对账、应收 | 财务核销 |
| 付款单 | 记录付款给供应商或物流商 | 付款单 | 供应商对账、物流对账、应付 | 财务核销 |
| 应收应付 | 汇总往来余额 | 应收列表、应付列表 | 对账单、收付款单 | 财务分析 |
| 合同管理 | 管理采购或销售合同 | 合同列表、详情 | 客户、供应商 | 订单、价格、对账 |

## 1. 采购管理

### 1.0 表单展示原则

采购相关表单统一按以下规则处理：

- `id` 作为系统主键隐藏，不在页面展示
- 除 `id` 外，其余表头字段、明细字段默认全部展示
- 当前阶段不做字段折叠、不做高级字段隐藏
- 编辑页、详情页字段保持一致，避免录入和查看口径不一致

### 1.1 采购订单

作用：

- 向供应商下达采购需求
- 记录采购商品、数量、价格、交货信息
- 作为采购入库和供应商对账的源头单据

建议表头字段：

- `id`
- `orderNo`
- `supplierId`
- `contractId`
- `orderDate`
- `status`
- `totalAmount`
- `remark`

建议明细字段：

- `materialId`
- `spec`
- `length`
- `quantity`
- `weightTon`
- `unitPrice`
- `amount`
- `remark`

建议状态：

- `草稿`
- `已审核`
- `部分入库`
- `已完成`
- `已关闭`

### 1.2 采购入库

## 3. 价格管理

### 3.1 价格核准

作用：

- 对已审核销售出库单进行最终价格和重量结算确认
- 控制销售出库最终成交价、结算重量、税额和备注
- 作为客户对账的前置业务单据，并将结果回写销售出库

建议字段：

- `id`
- `approvalNo`
- `depotHeadId`
- `organId`
- `deliveryDate`
- `totalWeight`
- `totalAmount`
- `status`
- `remark`

建议状态：

- `待核准`
- `已核准`

现版逻辑提取：

- 价格核准不是通用采购/销售价格审批模块，而是只针对已审核销售出库单
- 同一销售出库单只能生成一张价格核准单
- 创建时复制销售出库明细到核准明细，作为后续编辑和拆分基础
- 明细允许拆分多行，但每行都必须关联原始出库明细
- 核准确认后回写销售出库单头和出库明细：
  - 出库单头回写总金额、含税金额、价格核准标记、重量核准标记、备注
  - 出库明细回写重量、单价、金额、税额、备注
- 已核准记录如果已被客户对账单引用，则不能取消核准
- 销售出库重量变更后，核准单要同步刷新；若已核准，还需要继续回写出库金额

## 4. 物流管理

### 4.1 物流单

作用：

- 记录销售出库对应的承运信息和运费结算
- 连接销售出库、物流商、运费单价、总运费
- 为物流对账和运费付款提供依据

建议表头字段：

- `id`
- `billNo`
- `carrierId`
- `billTime`
- `unitPrice`
- `totalWeight`
- `totalFreight`
- `status`
- `deliveryStatus`
- `remark`

建议扩展字段：

- `vehicleNo`
- `driverName`
- `driverPhone`

建议状态：

- `未审核`
- `已审核`
- `未送达`
- `已送达`

现版逻辑提取：

- 物流单不是手工逐行录入商品，而是通过关联整张销售出库单生成业务范围
- 同一销售出库单不能重复挂到多个物流单
- 物流单详情页按关联销售出库单展开商品明细用于查看、导出、打印
- 物流单总重量来自关联销售出库单重量汇总，总运费按“总重量 × 单价”计算
- 查看详情时系统会按最新销售出库重量实时重算物流单重量和总运费
- 已审核物流单不能删除；已有付款记录的物流单不能删除或反审核

## 6. 对账管理

### 6.1 供应商对账单

作用：

- 汇总供应商采购、入库、付款记录
- 形成应付依据

建议数据来源：

- 采购订单
- 采购入库
- 付款单

核心结果：

- 本期采购金额
- 本期付款金额
- 期末应付余额

### 6.2 客户对账单

作用：

- 汇总客户已结算销售、出库、收款记录
- 形成应收依据

建议数据来源：

- 价格核准后的销售出库
- 收款单

核心结果：

- 本期销售金额
- 本期收款金额
- 期末应收余额

现版逻辑提取：

- 客户对账应以“价格核准后的销售出库明细”为口径，而不是原始销售出库
- 价格核准单一旦被客户对账单引用，价格核准单不能取消核准
- 因此前端设计时，客户对账单应放在价格核准之后，而不是直接放在销售出库之后

### 6.3 物流对账单

作用：

- 汇总物流单和付款记录
- 确认运费应付

建议数据来源：

- 物流单
- 付款单

核心结果：

- 本期运费金额
- 本期付款金额
- 期末应付余额

现版逻辑提取：

- 当前版本存在两套并行逻辑：
  - 运费对账汇总：按物流方汇总已审核物流单，并直接在物流单上记录付款状态和已付金额
  - 正式物流对账单：从未被其他对账单引用的已审核物流单生成对账单
- 正式物流对账单支持审核、签署、附件、已付金额累计
- 已签署的物流对账单不能删除；签署前必须先审核
- 前端菜单和交互应明确区分“运费对账汇总”和“物流对账单”，避免用户将两者视为同一单据

## 核心单据链路

### 采购链路

```text
供应商 / 商品 / 采购合同 / 价格核准
  -> 采购订单
  -> 采购入库
  -> 商品库存报表
  -> 供应商对账单
  -> 付款单
  -> 应付
```

### 销售链路

```text
客户 / 客户项目 / 商品 / 销售合同 / 价格核准
  -> 销售订单
  -> 销售出库
  -> 价格核准
  -> 物流单
  -> 商品库存报表
  -> 客户对账单
  -> 收款单
  -> 应收
```

### 物流链路

```text
物流商
  -> 物流单
  -> 物流对账单
  -> 付款单
  -> 应付
```

- 导出按钮

## 结论

本版系统按以下业务域推进：

- 采购管理
- 销售管理
- 价格管理
- 物流管理
- 库存报表
- 对账管理
- 财务管理
- 合同管理

后续前端设计、接口定义、权限拆分都建议按上述模块边界推进。
